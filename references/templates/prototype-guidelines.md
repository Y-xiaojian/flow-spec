# 原型设计通用规范（Prototype Guidelines）

> **性质**：与具体产品视觉无关的**通用**约定。交付到其他产品线时，通常**只需替换**《前端样式规范》文件（或路径），本文件可保持不变或仅做行业措辞微调。  
> **配套**：执行层细节见 `prototype-spec.md`；颜色、间距、组件 Token 一律以**当前项目**指定的《前端样式规范》为准（本仓库默认 `references/templates/frontend-style-guide.md`）。

---

## 1. 文档分层（必读）

| 层级 | 典型文件 | 是否通用 | 职责 |
|------|----------|----------|------|
| A. 前端样式规范 | 由项目提供，如 `frontend-style-guide.md` | **产品/项目定制** | 色板、字体、布局栅格、组件 CSS、交互控件命名与禁止项 |
| B. 原型通用规范 | 本文件 | **通用** | 交付物形态、页面类型、标注与数据、检查点、路径约定 |
| C. 原型模板 | `prototype-spec.md` | **通用** | 清单/标注/页面结构的可复制模板与字段约定 |

生成原型时：**先理解 B、C**，再严格按 **A** 落地样式与布局。

---

## 2. 目标与适用范围

- **目标**：在需求阶段产出可浏览器打开、可点击走查、可交给研发对齐的**静态原型**，并沉淀**字段/规则/接口**级说明，降低 PRD 返工。
- **适用**：中后台列表/表单/详情/审批/报表等 Web 管理端；不约束具体业务域。
- **非目标**：不等于视觉品牌稿、不等于生产代码脚手架；不要求特定前端框架。

---

## 3. 技术基线（通用最小集）

以下与「是否 Ant 风」无关，属于**可跨项目复用**的交付底线（若《前端样式规范》有更严要求，从其更严者）：

| 项 | 约定 |
|----|------|
| 交付形态 | 至少一个可直接打开的 **HTML**（或项目规定的等价单包产物） |
| 依赖 | **不依赖**外部 CDN/外链脚本样式字体图片；便于离线评审与归档 |
| 样式与脚本 | 与样式规范一致；常见做法为**内联**于单页，避免多文件散落 |
| 脚本能力 | 原生 JS 即可；仅用于页面切换、弹窗、列表筛选等演示 |
| 可访问性 | 原型阶段建议保留语义标签与可见焦点；细则可由样式规范补充 |

具体「侧边栏宽度、主色 hex、Tag 映射」等**一律不写在本文**，避免与定制样式规范冲突。

---

## 4. 页面类型与必备信息（通用）

每种类型至少包含：**主操作入口**、**与状态相关的样例**、**关键字段可辨识**。

| 类型 | 必备信息块（逻辑层） |
|------|----------------------|
| 列表 | 筛选/查询区、主列表、分页或等价导航、主要批量/新建操作 |
| 表单 | 分组逻辑、必填与校验提示占位、提交/取消/草稿（若业务需要） |
| 详情 | 主实体标识、状态、分区展示（基础信息/子表/时间线等按业务） |
| 审批 | 待办信息、通过/驳回/转交等操作区、审批轨迹或时间线 |
| 报表/看板 | 核心指标区、图表占位（须写清图表含义，禁止无说明空白） |

多页原型须说明**页面间跳转关系**（侧栏、按钮、链接）。

---

## 5. 标注与模拟数据（通用）

- **标注**：关键控件、字段、按钮应能追溯到规则（校验、权限、**单据状态**等业务规则）；推荐属性名 `data-annotation`（若样式规范规定其他名从其规定）。
- **同步产出**：与 HTML 同级的**标注说明文档**（如 `annotations.md`），结构见 `prototype-spec.md`。
- **模拟数据**：列表须覆盖**多种业务状态**（含异常/边界）；表单须有**合理预填**；禁止全空列表或全同状态充数。

---

## 6. 路径与命名（通用约定）

| 产物 | 约定路径模式 |
|------|----------------|
| 头脑风暴结论 | `temp/brainstorm/<需求名>.md` |
| 静态原型 | `temp/prototypes/<需求名>/index.html` |
| 标注说明 | `temp/prototypes/<需求名>/annotations.md` |

其他仓库可改 `temp/` 前缀，但须在项目 README 或技能 contract 中写死一处，避免混乱。

---

## 7. 与需求流水线的检查点（通用）

- **页面清单确认**：列出页面名、类型、核心字段与跳转后再生成 HTML。  
- **原型确认**：干系人浏览器走查后再进入功能点清单 / PRD。  

具体口令与下游技能名以项目 `workflows/write-doc.md` 为准。

---

## 8. 迁移到其他产品时怎么做

1. **保留**本规范 + `prototype-spec.md`（通用层）。  
2. **替换或 fork**《前端样式规范》：写入该产品的主色、布局、组件 Token 与禁止项。  
3. 在技能或 README 的 contract 中，把「样式规范路径」改为新产品文件路径。  
4. **不要**把产品专属 hex、侧边栏像素写回本文件，以免再次耦合。

---

## 9. 修订原则

- 本文件只增删**与视觉无关**的流程与结构约定。  
- 任何「颜色 / 字号 / 圆角 / 具体布局尺寸」的变更应在《前端样式规范》中完成。
