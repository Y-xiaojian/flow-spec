# 需求文档结构模板（SRM 八章）

> 与 `flow-spec:requirement-doc-writing` 配套：规定**章节骨架**与**第四章最小丰富度**。行业措辞、术语以 `srm-glossary.md` 为准。

---

## 文档信息（文首）

- 文档编号、版本、日期、状态、作者、审核人  
- 修订记录表

---

## 一、项目概述


| 小节        | 必备内容                                     |
| --------- | ---------------------------------------- |
| 1.1 背景与痛点 | 痛点表（痛点 / 业务代价 / 现有绕过方式），**与头脑风暴「WHY」一致** |
| 1.2 项目目标  | 可量化目标表（目标 / 衡量指标）                        |
| 1.3 目标用户  | 角色、描述、核心诉求                               |

## 四、功能需求详述（**固定层级结构**）

第四章采用 **「模块 → 业务介绍 → 功能点 → 概述 / 系统功能 / 页面」** 三级标题体系，与功能清单中的**模块**划分对齐。同一模块下的多个 FT 可拆成多个「功能点」小节，或合并为一个功能点但在标题中列出全部 **FT-xxx**。

### 标题层级示例

```markdown
## 四、功能需求详述

### 4.1 询价单管理

#### 业务介绍

（模块级说明，见下表「写什么」）

#### 4.1.1 询价单列表与筛选（FT-001 / FT-002 / FT-003）

##### 概述

##### 系统功能

##### 页面

#### 4.1.2 新建与提交询价（FT-004 …）

##### 概述

##### 系统功能

##### 页面

### 4.2 询价审核

#### 业务介绍

#### 4.2.1 …

…
```

---

### 4.x 【模块名】（`###` 三级标题）

#### 业务介绍（`####`，模块内仅出现一次）


| 要写什么    | 说明                                        |
| ------- | ----------------------------------------- |
| 模块定位    | 本模块在整体业务流程中的位置、解决什么问题                     |
| 与头脑风暴对齐 | 一句话指向 `changes/<CHANGE-ID>/brainstorm/<需求名>.md` 中相关痛点/场景 |
| 主要角色    | 谁使用本模块、与上下游模块关系                           |
| 本模块功能边界 | 做 / 不做的范围（可与第二章范围外呼应）                     |


篇幅建议：**1 个自然段 + 可选小表**，避免空泛口号。

---

#### 4.x.y 【功能点名称】（FT-aaa / FT-bbb …）（`####` 四级标题）

每个功能点下**固定三个五级标题**（`#####`），顺序如下。

##### 概述


| 要写什么 | 说明                         |
| ---- | -------------------------- |
| 功能目的 | 1～2 段话：用户要完成什么事、价值是什么      |
| 用户故事 | `作为…，我希望…，以便…`（与角色、痛点一致）   |
| 典型场景 | 2～4 步主流程（编号：角色、前置、操作、系统反馈） |


##### 系统功能


| 要写什么      | 说明                                          |
| --------- | ------------------------------------------- |
| 前置 / 后置条件 | 状态、权限、数据；不满足时的行为                            |
| 业务规则      | **BR-xxx** 列表，可判定、可审计                       |
| 校验规则      | **VR-xxx**，与字段/界面约束对应；可写「见系统功能表」避免重复        |
| 异常与边界     | 表：场景 / 系统行为 / 用户提示（含空列表、无权限、并发等）            |
| 通知与集成     | 若有：事件 / 渠道 / 接收方 / 要点                       |
| 权限        | 若有：角色 × 可见 / 可操作                            |
| 验收标准      | **AC-xxx** 或 Given/When/Then 表，可测试          |
| 数据与接口     | 按需：主实体字段、接口要点、与单据状态等业务规则的关系（**不以状态机图形式呈现**） |


##### 页面

用 **一段连续说明**（可含编号小步骤），指导读者：

1. 在浏览器中打开原型路径，例如 `changes/<CHANGE-ID>/prototypes/<需求名>/index.html`（或部署后的等价 URL）。
2. 如何通过侧栏 / 按钮进入本功能对应界面（写出页面名称或原型中的导航文案）。
3. 建议截取的画面范围（如：整页、列表区、弹窗、关键 Tab），以便与 **概述 / 系统功能** 对照评审。
4. 明确：**请需求负责人或产品将截图粘贴到本节下方**；PRD 中保留占位，例如空一行后写 `> **界面附图**：（请粘贴浏览器截图）`，或使用 `![界面示意](待粘贴)`（具体 Markdown 图语法按你们文档规范二选一即可）。

**不要求**在本节用完整「字段表」复制原型；与界面强相关的校验与枚举已在 **系统功能** 中用表写清即可。

---

### 写作注意

- **禁止**：仅有「概述」一段话 +「页面」无截图指引、**系统功能** 无 BR/验收/异常。  
- **FT 覆盖**：标题中写全本节覆盖的 FT 编号；合并写时，验收与异常须覆盖各 FT 差异。  
- **编号延续**：BR / VR / AC 全章连续编号，不 per 模块重置（与现有 PRD 习惯一致即可，但须统一）。

## 全文自检（生成第四章后过一遍）

- 每个 **4.x 模块** 是否都有 **「#### 业务介绍」**  
- 每个 **4.x.y 功能点** 是否都有 **概述 / 系统功能 / 页面** 三节，且 **系统功能** 中含异常与验收  
- **页面** 是否含**截图操作说明**与**贴图占位**  
- BR/VR/AC 与头脑风暴、功能点清单一致且无悬空编号
