# 功能点清单技能

## 前置条件

加载本技能前，以下产物必须已存在：

- `changes/<CHANGE-ID>/prototypes/<需求名>/index.html` — 已通过人工样式确认的原型  
- `changes/<CHANGE-ID>/prototypes/<需求名>/annotations.md` — 原型标注说明

若前置缺失，先执行 `flow-spec:prototype-design`。

## 输入 → 输出 contract


| 输入       | 来源                                     |
| -------- | -------------------------------------- |
| 确认后的原型标注 | `changes/<CHANGE-ID>/prototypes/<需求名>/annotations.md` |
| 头脑风暴结论   | `changes/<CHANGE-ID>/brainstorm/<需求名>.md`             |



| 输出           | 路径                                                                    |
| ------------ | --------------------------------------------------------------------- |
| **功能点清单（主）** | `changes/<CHANGE-ID>/requirements/<需求名>-feature-list.csv`                            |
| **清单说明（辅）**  | `changes/<CHANGE-ID>/requirements/<需求名>-feature-list.md`（元信息、统计、与 CSV 路径；**不写重复大表**） |


## CSV 格式（主交付物）

### 表头（首行，顺序固定）

```text
编号,系统,角色,模块,功能,功能类型,前端,后端,优先级
```


| 列名   | 含义           | 填写说明                                |
| ---- | ------------ | ----------------------------------- |
| 编号   | 追溯号          | `FT-001` 起全需求连续递增，与后续 PRD 第二章功能清单一致 |
| 系统   | 终端/系统        | 如 `SRM`、`SRM/PC`、`SRM/移动端`、`接口服务`   |
| 角色   | 用户角色         | 如 `采购员`、`采购主管`、`供应商联系人`、`系统`        |
| 模块   | 业务模块         | 与原型侧栏/域划分一致，如 `询价单管理`、`开放接口`        |
| 功能   | 功能点名称        | 可验收的一则能力；**能与模块合并时见下节**             |
| 功能类型 | 变更类型         | `新增功能` / `修改功能` / `优化` 等，与你们发布分类对齐  |
| 前端   | 详细描述（界面侧）    | 页面、列表字段、筛选、按钮、弹窗、权限可见性等；无则填 `-`     |
| 后端   | 详细描述（服务侧）    | 接口、校验、任务、通知、数据规则、与外部系统交互等；无则填 `-`   |
| 优先级  | P0 / P1 / P2 | 与既有约定一致                             |


### 模块 / 功能「能合并就合并」

1. **列级合并**：若某条只有模块级命名、无更细「功能」名，则 `功能` 填 `-`，或将「父模块 / 子能力」**只写在 `模块` 列**（例：`开放接口 / 代发货单据查询`），`功能` 填 `-`。
2. **行级合并**：同一模块下多条强相关细目可**合并为一行**：`功能` 用简短标题概括，在 `前端` 或 `后端` 内用 `1.` `2.` `3.` 列出子要求（与第二张示例图一致）。
3. **避免重复**：连续多行同一 `模块` 时，允许 `模块` 留空表示「与上一行相同」（**须在辅组 `.md` 中注明**）；若工具或流程不便于空单元格，则每行重复填写 `模块`。

### CSV 书写规则

- 编码：**UTF-8**（Excel 打开乱码时可用 UTF-8 BOM，按需）。  
- 分隔符：英文逗号 `,`。  
- 字段含逗号、双引号或换行：用双引号包裹，内部 `"` 转义为 `""`。  
- 换行：单元格内用真实换行时，整字段必须用双引号包裹。

### 辅组 `.md` 最小内容

```markdown
# <需求名> 功能点清单

- 创建日期：YYYY-MM-DD
- 状态：草稿 / 已确认
- **完整清单（CSV）**：`changes/<CHANGE-ID>/requirements/<需求名>-feature-list.csv`
- 关联原型：`changes/<CHANGE-ID>/prototypes/<需求名>/index.html`
- 关联头脑风暴：`changes/<CHANGE-ID>/brainstorm/<需求名>.md`

## 统计

| 优先级 | 行数 |
|--------|------|
| P0 | n |
| … | … |

> 若 CSV 中「模块」列存在空值继承上一行，以 CSV 为准解析。
```

## 执行步骤

### Step 1 — 从原型与标注提取

遍历 `annotations.md` 各页面：交互控件、筛选、列表、详情、弹窗、状态变化等均映射为行（或合并行中的编号列表）。

对照 `index.html` 的页面结构与 `data-annotation`，补全「前端」列；推断接口、异步任务、通知等写入「后端」列。

### Step 2 — 对齐头脑风暴

补充标注中未体现但头脑风暴已明确的接口类、规则类、权限类条目；`系统`/`角色` 与结论中的角色表一致。

### Step 3 — 编号与优先级

- `FT-001` 起连续编号，**不**按模块重置。  
- P0：主路径不可用则整体不可用；P1：重要体验；P2：可延期。

### Step 4 — 写入 CSV 与辅组 md

先写 **CSV**（表头 + 数据行），再写 `**.md`**（统计行数来自 CSV）。

在回复中输出摘要：

```text
功能点清单已生成：
- changes/<CHANGE-ID>/requirements/<需求名>-feature-list.csv（主）
- changes/<CHANGE-ID>/requirements/<需求名>-feature-list.md（说明）

共 <N> 行功能点，P0: <n>，P1: <n>，P2: <n>

请确认无误后输入「功能点确认」，将继续生成需求文档。
```

**⛔ CHECKPOINT：收到「功能点确认」后，才继续执行 `requirement-doc-writing`。**

## 避坑


| 情况         | 处理                                              |
| ---------- | ----------------------------------------------- |
| 功能点超过 40 行 | 检查是否应「行级合并」或拆期                                  |
| 前后端难以拆分    | 先写入更确定的一侧，另一侧 `-`，在 PRD 再细化                     |
| PRD 需要表格   | `requirement-doc-writing` 从 **CSV** 生成或同步第二章功能表 |
