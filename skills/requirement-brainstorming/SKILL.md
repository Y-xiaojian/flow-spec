# 从想法到设计（头脑风暴）

## 输入 → 输出 contract

| 输入 | 来源 |
|------|------|
| 用户的需求描述（自然语言） | 对话 |
| 项目上下文（可选） | 仓库文件、近期提交、已有文档 |

| 输出 | 路径 |
|------|------|
| 头脑风暴结论文档 | `changes/<CHANGE-ID>/brainstorm/<需求名>.md` |

结论文档必须包含：**核心场景**、**用户角色**、**主要操作路径**、**边界与排除项**、**选定方案及理由**。

---

## 概述

通过协作对话，把模糊想法变成可评审的设计与规格说明。

先理解当前项目上下文，再**一次只问一个问题**收敛需求；在理解要做什么之后，分块呈现设计并征得用户确认。

<HARD-GATE>
在用户批准设计之前，**不得**调用任何实现类技能、**不得**写代码、**不得**脚手架、**不得**做任何实现动作。无论看起来多简单都必须遵守。
</HARD-GATE>

## 反模式：「这太简单了不需要设计」

任何工作都要经过这一过程。待办清单、单文件小工具、改一行配置——全部如此。「简单」往往最容易因未澄清假设而返工。设计可以很短（真正简单时几句话即可），但必须呈现并获批。

## 检查清单（按顺序逐项完成）

1. **了解项目上下文** — 浏览文件、文档、近期提交  
2. **挖痛点（WHY）** — 先问业务问题，再问功能需求（见「toB 痛点三问」）
3. **澄清问题（WHAT）** — 一次一个问题，弄清目的、约束、成功标准  
4. **提出 2～3 种方案** — 说明取舍与你的推荐  
5. **呈现设计** — 按复杂度分块，每块征得用户认可  
6. **撰写结论文档** — 保存到 `changes/<CHANGE-ID>/brainstorm/<需求名>.md`  
7. **转入下一阶段** — 若走 write-doc 路线，调用 `flow-spec:prototype-design`；若用户只要文档链的后续某一环且输入已足，按其意图加载对应技能（如 `flow-spec:requirement-doc-writing`、`flow-spec:technical-doc-writing`），不得调用已移除的编码类技能

## toB 痛点三问（WHY 优先，禁止跳过）

<HARD-GATE>
toB 软件失败的首要原因：做了用户「要的」，而不是用户「需要的」。
澄清功能之前，必须先问清楚业务痛点。
</HARD-GATE>

在进入「要做什么」之前，优先问以下三个问题（一次一个）：

**Q1 — 现在哪里最痛？**
> 「现在的流程中，哪个环节最让你或你的团队头疼？」

**Q2 — 这个痛点的业务代价是什么？**
> 「这个问题带来了什么实际损失？（时间、错误率、人力、客户投诉……）」

**Q3 — 现在是怎么绕过这个问题的？**
> 「目前你们是怎么处理这个问题的？有没有临时方案或人工补丁？」

痛点三问的结论写入结论文档的「**业务痛点**」章节，作为所有后续设计决策的依据。

## 流程（含 CHECKPOINT）

```
探索上下文
  ↓
【WHY】toB 痛点三问（一次一个）
  ↓
⛔ CHECKPOINT 1：痛点已确认，用户输入「痛点确认」
  ↓
【WHAT】澄清问题（一次一个）
  ↓
⛔ CHECKPOINT 2：核心问题已全部澄清，用户输入「澄清完成」
  ↓
提出 2～3 种方案 + 推荐
  ↓
分块呈现设计（每块征得认可）
  ↓
⛔ CHECKPOINT 3：设计方案获得用户批准，用户输入「方案确认」
  ↓
写结论文档 → changes/<CHANGE-ID>/brainstorm/<需求名>.md
  ↓
⛔ CHECKPOINT 4：结论文档完整，用户输入「结论确认」
  ↓
转入下一阶段
```

**终点状态：** write-doc 路线下调用 `flow-spec:prototype-design`；仅做单点文档时由用户指定下一技能（如 PRD、技术说明）。本包不包含编码与实现计划技能。

## 执行要点

**理解需求时：**
- 先看当前仓库状态（文件、文档、近期提交）
- **先问 WHY（痛点），再问 WHAT（功能）**，禁止反转顺序
- 一次只问一个问题；需要展开时拆成多次提问
- 尽量用选择题降低回答成本；开放题亦可
- 聚焦：痛点代价、目的、约束、成功标准

**探索方案时：**
- 给出 2～3 种做法及取舍
- 先给出推荐项并说明理由

**呈现设计时：**
- 分块书写，复杂度高的块可写 200～300 字量级
- 每块后询问是否认可
- 覆盖：业务架构、模块边界、数据流、错误与边界、验收关注点（文档视角）
- 随时准备回退澄清

## 设计定稿之后

**产物（artifact）：** 将已定稿的结论写入 `changes/<CHANGE-ID>/brainstorm/<需求名>.md`，格式如下：

```markdown
# <需求名> 头脑风暴结论

## 业务痛点（WHY）
| 问题 | 业务代价 | 现有绕过方式 |
|------|---------|------------|
| ... | ... | ... |

## 核心场景
...

## 用户角色
| 角色 | 描述 |
|------|------|

## 主要操作路径
1. ...

## 选定方案
方案：... 理由：...

## 边界与排除
- 本期不做：...
```

**下一步：**
- write-doc 路线 → 调用 `flow-spec:prototype-design`
- 其它路线 → 按用户意图调用下一文档技能（须满足该技能的前置输入）

**不要**在此步骤调用任何代码实现或仓库外的未授权工具。

## 原则

- **WHY 先于 WHAT** — 痛点未确认前，不讨论功能  
- **一次一个问题** — 避免信息过载  
- **优先选择题** — 降低回答成本  
- **YAGNI** — 设计里砍掉非必要功能  
- **多方案比较** — 先比较再定案  
- **分块确认** — 呈现一块、确认一块  
- **保持灵活** — 不清楚就回去问
