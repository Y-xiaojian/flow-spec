# 技术设计文档编写

## 概述

编写**技术设计文档**时，须以**已冻结范围的需求/规格**为源头：设计边界、规则、验收口径不得超出上游文档。用**追溯表**将需求条目映射到设计章节与实现边界。

交付物中至少包含：**一张主流程图**（如 flowchart）与**至少一张时序图**（sequence），推荐 Mermaid，便于评审与版本管理。

## 执行前必读

- `references/templates/technical-doc-template.md`（若存在）或项目约定的技术设计模板
- 关联的**需求/规格**定稿版本
- 项目术语与 API 约定（若有）

## 执行步骤

1. **对齐需求** — 打开关联需求文档，固定版本与范围；列出须覆盖的需求编号或章节。  
2. **填写追溯表** — 在技术文档中建立「需求 → 设计章节」映射。  
3. **按模板撰写** — 背景、上下文、架构、接口、数据、错误与边界；**不**在文档中私自扩大需求未涵盖的功能（应回写需求或记入风险）。  
4. **流程图** — 主路径与关键分支。  
5. **时序图** — 核心链路至少一幅，标外部依赖与失败路径（如需要）。  
6. **保存** — 默认保存到 `flow-spec/temp/technical/`（见 `references/rules/storage.md`）；若业务仓库另有约定路径，在文首说明并与索引一致；变更记入附录或修订记录。

## 注意

- 接口需包含路径/方法或等价约定、参数、错误语义，并与仓库现有风格一致。  
- 数据设计需考虑迁移与兼容（若适用）。  
- 流程与时序须能被开发直接实现，与需求规则无矛盾。
